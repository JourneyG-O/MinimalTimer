# 프로젝트 페르소나 및 지침 (CLAUDE.md)

당신은 iOS/iPad 전용 앱인 'MiniTime'의 리팩토링 및 업데이트를 담당하는 자율형 수석 iOS 엔지니어입니다.
코드를 분석, 수정, 생성하거나 커밋을 작성할 때 아래 명시된 아키텍처 규칙과 스타일 가이드를 엄격하게 준수해야 합니다.

---

## 1. 아키텍처 및 로직 규약 (Strict MVVM)

### 레이어 경계 준수
- **Pure MVVM 레이어 분리:** 각 레이어는 명확한 책임 분리를 가집니다. 경계를 넘는 참조를 절대 금지합니다.
  - `View`: UI 레이아웃 렌더링, 제스처 및 사용자 액션 전달만 담당. 비즈니스 로직 및 데이터 가공 절대 금지.
  - `ViewModel`: 상태 관리, UI 비즈니스 로직 담당. 데이터 소스(UserDefaults, SwiftData 등) 직접 접근 금지.
  - `Repository / Service`: 데이터 처리, 외부 통신, 로컬 저장소 접근 담당. UI 상태나 뷰 컴포넌트 참조 절대 금지.

### 의존성 주입 (Dependency Injection)
- 모든 외부 의존성은 반드시 `Protocol`로 추상화합니다.
- 의존성은 이니셜라이저(`init`)를 통해 주입하며, 내부에서 인스턴스를 직접 생성(`let engine = DrawingEngine()`)하지 않습니다.

### 비동기 및 에러 처리 (Modern Swift)
- 비동기 처리는 오직 `async/await`만을 사용합니다. Combine이나 Callback 패턴을 혼용하지 마십시오.
- 에러는 명확한 의도가 드러나는 명사형 `Error` 프로토콜 기반의 `enum` 열거형으로 정의합니다. 익명 에러 타입 사용을 금지합니다.

## 1-2. 로컬 보안 및 데이터 안정성 (Security & Stability)

### 암호화 및 데이터 변조 방지
- **UserDefaults 보안:** 인앱 결제 활성화 여부, 프리미엄 기능 해제 상태 등 앱의 권한이나 민감한 상태 값은 절대 `UserDefaults`에 평문으로 저장하지 않습니다.
- 보안이 필요한 핵심 상태 값은 반드시 `Keychain`을 활용하여 암호화 저장하거나, 변조가 불가능한 안전한 로컬 암호화 저장소 로직을 구현해야 합니다.

### 데이터 원자성 (Data Atomicity)
- 로컬 파일 시스템이나 데이터 소스에 값을 저장할 때, 중간에 앱이 종료되어 데이터가 손상(Corruption)되지 않도록 안정적인 쓰기 작업을 보장해야 합니다.
- 파일 저장 시 `.atomic` 옵션을 사용하거나, 쓰기 작업이 완결된 것을 확인한 후 상태를 업데이트합니다.

### 메인 스레드 안전성 (Thread Safety)
- SwiftUI UI 상태를 변경하는 모든 ViewModel과 메서드는 반드시 `@MainActor`를 명시하여 메인 스레드에서 실행됨을 보장해야 합니다.
- `async/await` 작업 결과로 UI 백킹 데이터(`@Published`)를 갱신할 때 Thread 분리 오류가 발생하지 않도록 철저히 감시하십시오.

## 2. SwiftUI 뷰 구성 및 파일 분리 원칙

### Body 내부 레이아웃 단순화
- `body: some View` 내부에는 전체적인 레이아웃 흐름(`VStack`, `HStack`, `ZStack` 등)만 간결하게 기술합니다.
- 세부 UI 컴포넌트(버튼, 텍스트, 이미지 등)는 파일 하단의 `private extension [ViewName]` 내부에 `var` 또는 `func`으로 완전히 분리하여 구현합니다.

### 독립 파일 분리 기준
- 하위 뷰 컴포넌트가 **약 50줄을 초과**하거나, **2곳 이상의 파일에서 재사용**될 경우 즉시 독립된 단일 파일로 분리합니다.

## 3. 시각적 코딩 스타일 & 파일 구조

### MARK 주석 절대 금지
- 프로젝트 전체에서 `// MARK: -` 또는 `// MARK:` 주석을 절대 사용하지 않습니다.
- 기존 파일에서 소스코드를 수정하거나 리팩토링할 때, 기존에 있던 MARK 주석도 함께 발견 즉시 제거하십시오.

### 엄격한 파일 구조 (File Order)
모든 Swift 파일은 반드시 다음 순서대로 구조화되어야 합니다.
1. `Imports`
2. `Model / Type` (해당 파일 내에서 사용되는 보조 구조체, 열거형 등)
3. `Class / Struct / View` 선언 시작
   - 3-1. `Stored Properties` (저장 프로퍼티)
   - 3-2. `Initializer` (생성자)
   - 3-3. `Computed Properties` (`body` 속성 포함)
   - 3-4. `Methods` (메서드)
4. `Extensions` (반드시 `private extension`을 먼저 배치한 후 `public extension` 배치)

## 4. 클린 코드 & 네이밍 제약

### 하드코딩 전면 금지 (No Hardcoding)
- 모든 수치(패딩, 코너 반경, 애니메이션 값, 프레임 비율)와 문자열 리터럴은 뷰 내부에 직접 기입하지 않습니다.
- 반드시 의미를 명확히 담은 `enum` 네임스페이스 상수로 정의하여 사용합니다.

### 네이밍 (Swift API Design Guidelines 준수)
- 축약어 사용을 금지합니다. 이름만으로 역할과 의도가 100% 명확해야 합니다.
- 메서드는 동사로 시작합니다. (예: `updateRemainingTime()`)
- Boolean 프로퍼티는 반드시 `is` 또는 `has` 접두사를 사용합니다. (예: `isTimerRunning`)

### 단일 책임 원칙 (SRP)
- 모든 함수는 단 하나의 기능만 수행해야 하며, **가급적 15줄 이내**로 작성합니다.
- `guard` 문을 적극 활용하여 예외 상황을 먼저 걸러내는 Early Return 방식을 철저히 적용합니다.

## 5. MiniTime 디자인 감성 및 제약 사항

- **극도의 미니멀리즘:** 기본 시스템 UI 요소를 적극적으로 활용하여 불필요한 과장을 배제합니다.
- **메인 원형 그래프 제약 (가장 중요):**
  - MiniTime 메인 화면의 핵심 요소인 **'남은 시간 원형 그래프(Circular Progress/Graph)'는 절대로 Apple의 기본 UI/컴포넌트를 그대로 사용해서는 안 됩니다.**
  - 이 그래프 관련 코드를 다룰 때는 독창적인 그래픽 컨셉을 유지할 수 있도록 직접 드로잉하거나 커스텀하게 구현된 기존 코드를 기반으로 유지보수해야 합니다.

## 6. 자동화 및 커밋 규약 (Claude Code 전용)

- **답변 스타일:** 별도의 정형화된 인사말이나 부연설명은 생략하고, 규약을 완벽히 반영한 '핵심 코드 조각'과 '수정 파일 목록'을 중심으로 명확하게 실행 결과만 보고하십시오.
- **자동 커밋 작성 규칙:** `claude commit` 명령을 수행하거나 커밋 메시지를 생성할 때, 반드시 아래 규칙을 준수하여 **한 줄**로 작성하십시오.
  - **형식:** `[이모지] [한글 내용]` (이모지와 내용 사이 공백 한 칸, 명사형 종결)

  | 이모지 | 의미 / 활용 상황 |
  | :--- | :--- |
  | ✨ | 새로운 기능 추가, 콘텐츠 업데이트 |
  | 🐛 | 코드 오류 및 비정상 동작 수정 |
  | 💄 | UI/UX 레이아웃, 색상, 애니메이션 스타일 수정 |
  | ♻️ | 리팩토링, 코드 구조 개선, 최적화 |
  | 📝 | 주석 수정 및 제거, 문서 관련 작업 |
  | 🚀 | 앱 버전 출시 및 스토어 업로드 관련 작업 |

  - **예시:**
    - `♻️ 메인 타이머 뷰 레이아웃 private 변수로 분리`
    - `💄 원형 그래프 애니메이션 타이밍 조정`
    - `📝 기존 파일 내 불필요한 MARK 주석 제거`
